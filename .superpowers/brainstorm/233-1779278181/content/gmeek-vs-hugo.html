Gmeek vs Hugo：是否需要更换框架？

分析两个框架在自定义背景、音乐播放器、玻璃质感主题方面的差异

当前架构：Gmeek

Gmeek 通过 GitHub Actions 从 Issues 生成静态 HTML。每个页面是完全独立的 HTML 文件，CSS/JS 全部内联。Gmeek.py 脚本生成时会在 <head> 和 <style> 中插入内容。

自定义方式：在 blogBase.json 中配置 head、style、script 等字段，Gmeek 构建时将这些内容注入每个页面。

核心问题：实现你的目标需要什么？

需求 Gmeek 能否实现 Hugo 能否实现
全局背景图片 通过 style 字段注入 CSS 模板中直接编写
玻璃质感 (glassmorphism) 纯 CSS 实现，注入即可 纯 CSS 实现
音乐播放器按钮 script 字段注入 JS + HTML 模板 partial 引入
响应式布局 CSS media queries CSS media queries
主题色 (天蓝) 覆盖 Primer CSS 变量 模板变量 + CSS
Issue 自动构建 原生支持 需额外配置
A

保留 Gmeek，通过注入自定义

利用 blogBase.json 的 head/style/script 字段注入 CSS 和 JS，实现所有前端效果。保持 Issue → 自动构建的工作流不变。

B

迁移到 Hugo

完全自定义模板系统，更灵活但需要：迁移所有文章、配置 GitHub Actions 构建 Hugo、放弃 Issue 撰写方式。

方案 A 优势

  • 零迁移成本，5 分钟即可开始写 CSS
  • 保持现有的 Issue 写文章流程
  • 所有前端效果（背景、glass、音乐）纯 CSS/JS 即可实现
  • GitHub Actions 自动构建已配置好

方案 A 局限

  • CSS/JS 通过 JSON 字段注入，修改需编辑 JSON（需注意转义）
  • 模板结构由 Gmeek 控制，无法改变 HTML 骨架
  • Gmeek 生成的每页 HTML 结构固定
